# Kiro を活用した開発の進め方

## 概要

Kiro はスペック駆動開発（Spec-Driven Development）をサポートするAI開発環境です。アイデアを要件→設計→実装タスクへと段階的に具体化し、AIと協働して開発を進めます。

## 開発フロー全体像

```
アイデア → スペック種類選択 → ワークフロー選択 → 要件定義 → 設計 → タスク生成 → 実装
```

---

## 1. スペック作成を開始する

Kiro のチャットで「〇〇を作りたい」「〇〇のバグを直したい」と伝えると、スペック作成が始まります。

### スペック種類の選択

最初に以下の選択肢が提示されます:

| 選択肢 | 説明 | 適用場面 |
|--------|------|---------|
| **Build a Feature** | 新機能を構築する | 新しい機能やモジュールを追加したいとき |
| **Fix a Bug** | バグを修正する | 既存の不具合を修正したいとき |
| **Quick Plan** | 素早くタスクリストを生成する | レビューなしで一気に計画を立てたいとき |

---

## 2. ワークフローの選択（Feature の場合）

Feature を選んだ場合、次にワークフローの開始点を選びます:

### Requirements-First（推奨）

```
要件定義(requirements.md) → 設計(design.md) → タスク(tasks.md)
```

- ビジネス要件が明確なとき
- 「何を作るか」が分かっていて「どう作るか」を検討したいとき

### Design-First

```
設計(design.md) → 要件定義(requirements.md) → タスク(tasks.md)
```

- 技術アプローチが明確なとき
- 既存システムのリファクタリングやアーキテクチャ変更のとき

---

## 3. 要件定義フェーズ

### やること

- Kiro がユーザーの入力をもとに `requirements.md` を生成
- ユーザーストーリーと受け入れ基準（EARS形式）で要件を構造化
- 生成後、Kiro が自動的に各要件を詳細化（Detailing）

### ポイント

- 生成された要件を確認し、不足や修正があればフィードバックする
- 「Analyze Requirements」リンクで曖昧な点の分析・改善ができる
- 要件が固まるまで何度でもやり取り可能

---

## 4. 設計フェーズ

### やること

- 要件に基づき `design.md` を生成
- アーキテクチャ、コンポーネント、データモデル、正当性プロパティを定義

### 含まれる内容

| セクション | 内容 |
|-----------|------|
| Overview | 技術スタック、設計方針 |
| Architecture | システム構成図（Mermaid） |
| Components and Interfaces | API・インターフェース定義 |
| Data Models | ER図、スキーマ |
| Correctness Properties | プロパティベーステストで検証する正当性条件 |
| Error Handling | エラーレスポンス設計 |
| Testing Strategy | テスト方針 |

### ポイント

- 正当性プロパティ（Correctness Properties）は実装後のプロパティベーステストの基盤になる
- 設計に問題があればこの段階で修正する

---

## 5. タスク生成フェーズ

### やること

- 設計に基づき `tasks.md` を生成
- 実装の順序と依存関係を定義した実行可能なタスクリストを作成

### タスクの構造

各タスクには以下が含まれます:
- タスクの説明と実装内容
- 受け入れ基準
- 依存するタスク
- ステータス（`pending` → `in_progress` → `completed`）

---

## 6. 実装フェーズ

### タスクの実行方法

- **個別実行**: タスクを1つずつ選んで実行
- **一括実行**: 「Run All Tasks」ボタンで全タスクをキューに投入

### 実行中のポイント

- Kiro がタスクに従ってコードを生成・修正
- 各タスク完了後にテストが実行される（Hook設定による）
- 問題があれば途中で中断し、設計やタスクの見直しが可能

---

## 7. セッションの種類

Kiro には2つのセッションタイプがあります:

| セッションタイプ | 説明 |
|----------------|------|
| **Spec** | 構造化されたスペック駆動開発（要件→設計→タスク） |
| **Vibe** | 会話的なQ&Aや探索的コーディング |

---

## 8. 自律性モード

| モード | 説明 |
|--------|------|
| **Autopilot**（デフォルト） | Kiro が自律的にタスクを完了する。変更の確認・取り消しはいつでも可能 |
| **Supervised** | ファイル編集ごとに承認を求める。変更をhunk単位で承認/拒否できる |

---

## 補助機能

### Steering（`.kiro/steering/`）

チーム標準やプロジェクト固有のルールをエージェントに追加指示として渡す仕組み。

- `inclusion: auto` — 常に適用
- `inclusion: fileMatch` + `fileMatchPattern` — 特定ファイル読み込み時に適用
- `inclusion: manual` — チャットで `#` 指定時に適用

### Hooks（`.kiro/hooks/`）

IDEイベントに応じた自動アクション。例:
- ファイル保存時にリント実行
- タスク完了後にテスト実行
- プロンプト送信前にルールチェック

### MCP（`.kiro/settings/mcp.json`）

Model Context Protocol サーバーの設定。外部ツール・APIとの連携に使用。

## 実践的なTips

1. **要件は日本語で書いてOK** — Kiro は日本語での要件定義・設計に対応
2. **フィードバックは具体的に** — 「もっと詳しく」より「認証エラー時のレスポンス形式を追加して」が効果的
3. **スペックは修正できる** — 後から要件や設計を変更し、タスクを再生成できる
4. **正当性プロパティを活用** — 設計段階で定義した性質がテストの指針になる
5. **段階的に進める** — 一度に全部作ろうとせず、フェーズごとに確認しながら進める

---

## クイックスタート

1. Kiro チャットで機能の概要を伝える
2. 「Build a Feature」を選択
3. 「Requirements」を選択（推奨）
4. 生成された要件を確認・修正
5. 「Generate Tech Design」リンクをクリック
6. 設計を確認・修正
7. 「Generate Task List」リンクをクリック
8. タスクを実行して実装開始
